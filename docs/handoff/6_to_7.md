# 6号窗口交接文档：统一问答编排层 / Tool Calling 增强链

## 1. 本窗口目标回顾

6号窗口的核心目标，是把项目中的问答能力从“第四周能跑的统一入口”升级为正式的企业级问答编排层。

本窗口不负责重新判定异常，不负责重写 RAG 检索，不负责人工复核闭环，而是负责把前面窗口已经完成的能力统一编排起来。

上下游关系如下：

- 4号窗口：负责 `audit_result`、`rule_hit`、结果层事实。
- 5号窗口：负责 `retrieval_service`、`rule_explanation_service`、`evidence`、`citation`。
- 6号窗口：负责 `/ask` 主链编排和 `/ask-lc` 增强链。
- 7号窗口：负责人工复核闭环。

6号窗口的最终目标是：

- `/ask` 作为正式主链，要求稳定、受控、可验收。
- `/ask-lc` 作为 LangChain / Tool Calling 增强链，可以展示能力，但不替代主链。

- `use_vector = true`：使用 `vector`

### 2.4 `/ask` 正式接入 5号窗口解释链

原本 explanation route 依赖旧 CSV 样本选择逻辑。

现在已升级为优先走正式解释链：

- `app.rag.rule_explanation_service.explain_audit_result_simple`

正式解释链固定为：

- `audit_result`
- `rule_hit`
- `rule_definition`
- `rule_chunk`
- `evidence`
- `citation`

当请求中包含 `audit_result_id`，或同时包含 `clean_id + anomaly_type` 时，优先走正式解释链。

如果没有结构化 ID，则保留旧 explanation fallback 兼容链，避免原有演示能力断掉。

### 2.5 mixed route 优化

原本 mixed route 会直接拿原始问题做规则检索，容易命中平台字段、价格清洗等泛规则。

现在已增加：

- `build_mixed_retrieval_query()`

它会根据 `analysis_result` 自动生成更聚焦的规则检索问题。

例如原问题：

- 先统计当前异常情况，再结合规则写一段简短汇报

会转换为：

- 低价异常规则、跨平台价差异常规则、规格识别风险规则分别是什么？业务人员复核时应关注什么？

这样 mixed route 更贴近业务汇报场景。

### 2.6 `/ask-lc` mixed 对齐主链

`/ask-lc` 中的 mixed 问题仍然强制走：

- `build_report_tool`

并已对齐 `/ask` 主链：

- analysis
- focused retrieval query
- retrieval_service
- report_tools

也就是说，增强链不是另起炉灶，而是复用同一套业务能力。

## 4. `/ask` 主链设计

### 4.1 主链入口

入口函数：

- `run_ask(req: AskRequest) -> AskResponse`

主链流程：

1. 校验 `question`。
2. 选择 `retrieval_mode`。
3. 使用 route classifier 判断问题类型。
4. 根据 route 调用对应受控流程。
5. 构造 `AskResponse`。
6. 写入 `ask_log`。
7. 返回结果。

---

## 5. Route 分类口径

当前支持 5 类 route：

- `analysis`
- `retrieval`
- `explanation`
- `mixed`
- `unknown`

### 5.1 analysis

适用问题：

- 统计类问题
- 数量类问题
- 占比类问题
- 平台 / 品牌分析类问题
- 近 7 天类问题

示例：

- 当前异常情况统计一下
- 近7天哪个平台异常低价最多？
- 哪个品牌跨平台价差最大？

调用链：

- `route_classifier`
- `analysis_tools`
- `AskResponse`

### 5.2 retrieval

适用问题：

- 规则类问题
- 依据类问题
- 定义类问题
- 处理口径类问题
- 复核流程类问题

示例：

- 低价异常规则是怎么判断的？
- 规格识别风险规则是什么？
- 业务人员看到异常后应该怎么复核？

调用链：

- `route_classifier`
- `retrieval_service.search_rules_simple`
- `AskResponse`

默认模式：

- `hybrid`

### 5.3 explanation

适用问题：

- 为什么某条商品被判异常？
- 为什么 `audit_result_id=xxx` 命中某规则？
- 为什么 `clean_id=xxx` 是规格风险？

正式链路：

- `route_classifier`
- `rule_explanation_service.explain_audit_result_simple`
- `AskResponse`

正式链触发条件：

- `audit_result_id` 不为空

或：

- `clean_id + anomaly_type` 不为空

fallback 链路：

- `route_classifier`
- `explanation_tools_fallback`
- `AskResponse`

fallback 仅用于兼容旧演示，不是推荐主链。

### 5.4 mixed

适用问题：

- 先统计，再结合规则写汇报。
- 先分析，再总结。
- 先找异常，再给复核建议。

调用链：

- `route_classifier`
- `analysis_tools`
- `build_mixed_retrieval_query`
- `retrieval_service.search_rules_simple`
- `report_tools.build_brief_report`
- `AskResponse`

特点：

- mixed route 不交给自由 Agent。
- mixed route 是受控编排。

### 5.5 unknown

适用问题：

- 未命中明确业务路由的问题。

当前兜底：

- `ask_llm`

注意：

- unknown 不是核心业务主链。
- 后续可以考虑改为“提示用户换一种业务问题问法”，避免泛聊天化。

## 6. Retrieval 服务接入说明

6号窗口正式使用：

- `app.rag.retrieval_service.search_rules_simple`

核心参数：

- `query`
- `top_k`
- `retrieval_mode`
- `rule_code`
- `rule_version`
- `anomaly_type`
- `rerank_enabled`

当前 `/ask` route 使用方式：

retrieval route：

- 直接用用户问题检索。

mixed route：

- 先根据 `analysis_result` 生成更聚焦的 `retrieval_query`，再检索。

explanation formal route：

- 由 `rule_explanation_service` 内部带 `rule_code`、`rule_version`、`anomaly_type` 强约束检索。

当前默认：

- `retrieval_mode = hybrid`
- `rerank_enabled = false`

说明：

- hybrid = baseline + vector
- baseline 负责 metadata 强约束和高精度匹配
- vector 负责自然语言语义补召回
- rerank 当前只是预留，未接真实 rerank 模型

## 9. Trace 设计说明

`/ask` 返回中包含 trace。

典型 analysis trace：

- step 1：`route_classifier`
- step 2：`analysis_tools`

典型 retrieval trace：

- step 1：`route_classifier`
- step 2：`retrieval_service`

典型 formal explanation trace：

- step 1：`route_classifier`
- step 2：`rule_explanation_service`

典型 mixed trace：

- step 1：`route_classifier`
- step 2：`analysis_tools`
- step 3：`retrieval_service`
- step 4：`report_tools`

trace 的作用：

- 方便调试。
- 方便验收。
- 方便面试解释调用链。
- 避免只看到最终 answer，不知道中间发生了什么。

## 11. 当前已知小限制

### 11.1 mixed route 当前只做单次规则检索

现在 mixed 会根据 `analysis_result` 生成一个聚焦 query，然后检索 top_k。

例如：

- 低价异常规则、跨平台价差异常规则、规格识别风险规则分别是什么？业务人员复核时应关注什么？

这可能导致某一类规则分数过高，例如跨平台价差人工复核规则排在最前。

后续可增强为：

- low_price 单独检索 1 条
- cross_platform_gap 单独检索 1 条
- spec_risk 单独检索 1 条
- 再统一交给 report_tools 汇总

但当前不建议在 6号窗口继续扩展，避免破坏刚稳定的主链。

### 11.2 explanation fallback 仍是旧链

没有 `audit_result_id` 或 `clean_id + anomaly_type` 时，仍会走旧 CSV fallback。

这是为了兼容旧演示问题：

- 为什么这个商品会被判成低价异常？

后续建议：

- 7号窗口或后续版本中，引导用户在解释类问题中携带 `audit_result_id`。
- 或者在 UI 中点击某条异常后自动带入 `audit_result_id`。

### 11.3 rerank 仍为预留

当前 `retrieval_service` 支持：

- `rerank_enabled`

但实际策略是：

- noop rerank

也就是说，当前没有接真实 rerank 模型。

这是 5号窗口预留能力，6号窗口只负责透传和保留扩展点。

7号窗口如果做人工复核，应基于已有：

7号窗口不应该做：
