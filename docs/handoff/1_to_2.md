## 二、本窗口目标回顾

1号窗口的目标不是开发具体业务能力，也不是正式设计数据库字段，而是将旧项目从：

> 原型阶段按周堆叠结构

升级为：

> 后续可持续维护的正式工程骨架。

为 2~8 号窗口提供：

- 统一目录；
- 统一落点；
- 统一交接基础。

本窗口应完成的核心任务包括：

- 重构项目目录树；
- 明确代码层与数据资产层边界；
- 拆分 API / orchestrator / rag / schemas / repositories / services / tools / workflows 等基础目录；
- 为 docs / tests / scripts / alembic / app_ui 预留正式位置；
- 在尽量不打散旧主链的前提下，完成旧系统资产迁移与兼容保留。

以上目标目前已基本完成。

当前仓库目录已经体现出新版骨架结构。

## 五、关键设计决策

---

### 决策 1：先重构骨架，不抢写数据库主体

1号窗口没有正式定义：

- MySQL 表结构；
- ORM 模型；
- 迁移脚本；
- review 状态机；
- RAG 证据 schema。

原因不是忽略这些内容，而是遵守窗口职责边界：

> 1 号窗口只负责搭工地，不负责提前施工数据库底盘和业务模型。

---

### 决策 2：旧系统资产采用“收编 + 兼容”策略，而不是暴力推翻

旧项目已有可运行主链，包含：

- ask；
- RAG；
- SQLite；
- 日志；
- 评测；
- 前端演示。

因此本窗口没有采取“全部删除重建”的方式，而是选择：

- 主体逻辑迁入新目录；
- 旧路径尽可能改为兼容层；
- 确保主链不断。

这样做的目的是降低升级风险，防止出现：

> 结构是新了，但主链跑不起来。

旧项目交接文档也明确提醒，升级最怕把现有主链打散。

---

### 决策 3：严格区分“代码层”和“运行资产层”

旧项目里 `app/data/` 和根目录 `data/` 的语义冲突明显。

本窗口已将方向调整为：

| 目录      | 职责                              |
|---------|---------------------------------|
| `app/`  | 放正式代码                           |
| `data/` | 放样本、处理结果、规则文档、索引、输出日志、旧数据库等运行资产 |

这项区分对 2 号窗口尤其重要。

未来数据库表、迁移、种子数据、导出物、审计日志都不能再混放。

---

### 决策 4：保留 `legacy_sqlite.py` 作为过渡数据底盘

旧 SQLite 能力当前仍有价值，它承载了：

- 第二周结果层事实；
- 部分查询能力。

因此本窗口没有删除 SQLite，而是将其收编为：

```text
legacy_sqlite.py
```

并明确其身份是：

> 旧系统兼容层 / 过渡数据底盘，不是未来正式数据库主实现。

这为 2 号窗口做：

```text
旧结果层 → 新正式数据模型
```

的映射提供了过渡抓手。

## 十、风险与注意事项

---

### 风险 1：把兼容层当成正式主体层

目前项目中仍存在部分兼容文件。

2 号窗口开始时，要分清楚：

- 哪些是未来正式实现；
- 哪些只是为了旧主链不断而保留的兼容层。

避免把兼容层继续越写越厚。

---

### 风险 2：直接照搬旧 SQLite 表，而不做业务建模

旧 SQLite 适合原型验证，不适合直接视作未来正式数据库方案。

2 号窗口要做的是：

> 业务建模。

不是：

> 把 SQLite 翻译成 MySQL。

---

### 风险 3：数据库设计脱离结果层事实

正式表设计必须尊重当前结果层已经存在的关键业务字段。

否则后面：

- 问答；
- 解释；
- 审核链；

都会出现：

> 数据库说东，结果层说西。

旧 SQLite 查询逻辑已经反映出这层依赖。

---

### 风险 4：2号窗口越界去重写 4~7 号窗口内容

2号窗口重点是：

> 数据库与数据模型。

不要顺手把以下内容全写进去：

- 规则引擎；
- RAG 检索；
- 问答编排；
- 人工复核界面。

否则又会回到窗口串台问题。
