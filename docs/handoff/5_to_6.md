# 5号窗口 → 6号窗口正式交接文档

## 1. 交接背景

5号窗口负责“规则解释 / RAG 检索解释层”的正式化建设。

本窗口承接
4号窗口已经完成的异常判定结果层，不重新判定异常，不重写规则引擎，不绕开 `audit_result / rule_hit / rule_definition`
自己解释异常原因。

5号窗口的核心任务是：

```text
把规则文档切成可检索 rule_chunk
把 rule_chunk 补齐 metadata
提供 baseline / vector / hybrid 检索能力
提供 rerank 预留
提供 retrieval_service 统一检索入口
提供 rule_explanation_service 规则解释入口
输出 evidence / citation
为 6号窗口 /ask 编排层提供稳定能力
```

最终解释链路固定为：

```text
audit_result -> rule_hit -> rule_definition -> rule_chunk -> evidence -> citation
```

### 3.7 检索器

已实现三个 retriever：

```text
app/rag/retrievers/baseline_retriever.py
app/rag/retrievers/vector_retriever.py
app/rag/retrievers/hybrid_retriever.py
```

#### baseline retriever

定位：

```text
metadata 感知检索器
读取数据库 rule_chunk
使用 rule_code / anomaly_type / section_title / chunk_type / keywords_json / metadata_json 打分
输出 RetrievalResult
带 score_reasons
```

适合：

```text
解释型场景
规则编码明确场景
需要高稳定性的规则检索
```

#### vector retriever

定位：

```text
FAISS + Ollama embedding 语义检索器
读取 data/faiss 索引
输出 RetrievalResult
保留 metadata
```

适合：

```text
自然语言规则问答
FAQ 召回
用户表达和规则文档措辞不完全一致的场景
```

#### hybrid retriever

定位：

```text
baseline + vector 最小混合检索
baseline 负责强约束
vector 负责语义补召回
融合后输出 RetrievalResult
```

当前融合权重：

```text
baseline_weight = 0.65
vector_weight = 0.35
```

说明：

```text
当前 hybrid 是最小可用实现，不是复杂调参版。
已经满足 5号窗口交付要求。
```

## 6. ExplanationSchema 输出结构说明

`ExplanationSchema` 是 6号窗口编排自然语言回答时最重要的输入。

结构示意：

```json
{
  "audit_result_id": 259,
  "clean_id": 52,
  "anomaly_type": "spec_risk",
  "final_summary": "...",
  "rule_facts": {},
  "evidences": [],
  "citations": [],
  "trace_notes": []
}
```

### 6.1 final_summary

确定性摘要，由 5号窗口生成。

6号窗口可以直接展示，也可以作为回答开头。

但注意：

```text
6号窗口可以润色表达
不能改变其中事实
不能让模型重新判断异常
```

---

### 6.2 rule_facts

结果层事实摘要。

来自：

```text
audit_result
rule_hit
```

包含：

```text
audit_result_id
clean_id
anomaly_type
is_hit
hit_rule_code
hit_rule_version
reason_text
hit_messages
input_snapshot
threshold_snapshot
computed_value
```

6号窗口回答“为什么判异常”时，必须优先使用 `rule_facts`。

---

### 6.3 evidences

内部证据链。

可能包含：

```text
audit_result evidence
rule_hit evidence
rule_definition evidence
rule_chunk evidence
```

6号窗口可以用于：

```text
trace 展示
debug
结构化响应
工具调用结果展示
```

不建议直接完整展示给普通用户，因为字段较多。

---

### 6.4 citations

对外展示引用。

用于：

```text
/ask 回答中的引用来源
UI 中规则依据展示
报告中的规则引用
```

citation 字段：

```text
citation_id
evidence_id
doc_title
section_title
section_path
chunk_id
source_doc_path
quoted_preview
citation_note
rule_code
rule_version
anomaly_type
```

6号窗口展示规则依据时，优先使用 `citations`，不要自己从 `chunk_text` 里再截。

---

### 6.5 trace_notes

解释过程追踪说明。

用于：

```text
6号窗口 trace
debug
smoke test
面试演示
```

它会记录：

```text
解释链路
是否使用 hybrid
baseline/vector 返回数
是否启用 rerank
是否发生降级
```

## 9. 6号窗口应该怎么接

6号窗口职责预计是：

```text
统一 /ask 编排
问题路由
analysis / retrieval / explanation / mixed 流程整合
统一响应结构

工具调用组合
```

6号窗口应把 5号窗口作为底层能力调用，而不是重写底层 RAG。

---

### 9.1 retrieval 类问题

用户问规则、流程、FAQ，例如：

```text
低价异常是怎么判断的？
跨平台价差规则是什么？
规格识别风险什么情况下命中？
业务人员看到异常后怎么复核？
```

6号窗口建议调用：

```python
from app.rag.retrieval_service import search_rules_simple
from app.rag.schemas import RetrievalMode

response = search_rules_simple(
    query=user_question,
    top_k=5,
    retrieval_mode=RetrievalMode.HYBRID,
    rerank_enabled=False,
)
```

然后用：

```text
response.results
response.trace_notes
```

组织回答。

---

### 9.2 explanation 类问题

用户问某条异常为什么被判定，例如：

```text
audit_result_id=259 为什么被判规格风险？
clean_id=52 为什么命中 SPEC_RISK？

    audit_result_id=audit_result_id,

```

然后优先使用：

```text

explanation.citations
explanation.trace_notes
```

---

### 9.3 mixed 类问题

如果用户问：

```text

```

6号窗口可以：

```text
先调用 analysis 工具统计整体情况
再选择 audit_result_id
再调用 rule_explanation_service 解释典型样本

```

注意：

```text

但规则解释细节仍应调用 5号窗口服务
不要在 6号窗口手写一套解释链
```

## 10. 6号窗口不能越界的内容

6号窗口不应做以下事情：

```text
不应重新判定异常
不应绕开 audit_result / rule_hit / rule_definition
不应直接根据 rule_chunk 文本反推出异常原因
不应自己重新定义 evidence / citation schema
不应重写 baseline / vector / hybrid retriever
不应用大模型输出覆盖 rule_facts
不应把 citation 当成内部 evidence 使用
不应让自由 Agent 自己决定是否读取 rule_hit
```

解释型问题必须坚持：

```text
先结果事实
再规则定义
最后文档引证
```

也就是：

```text
audit_result -> rule_hit -> rule_definition -> rule_chunk
```

## 11. 当前已知小问题 / 后续可优化点

### 11.1 audit_result evidence 的 computed_value 偏低价字段

当前 `build_audit_result_evidence()` 中的 `computed_value` 主要是低价字段模板：

```text
explicit_low_price_threshold
group_avg_price
price_to_group_avg_ratio
low_price_rule_source
```

对于 `cross_platform_gap` 和 `spec_risk`，更完整的计算值已经在：

```text
rule_facts
rule_hit evidence
```

中存在。

所以当前不影响解释链通过。

后续可优化为：

```text
按 anomaly_type 构建 audit_result evidence computed_value
```

这不是 6号窗口阻塞项。

---

### 11.2 rerank 当前只是预留

当前没有接真实 rerank 模型。

状态：

```text
rerank_enabled = False
NoopReranker
score_reasons 追加 rerank_disabled
```

后续如果要接真实 rerank，应另开任务，不建议 6号窗口直接改。

---

### 11.3 hybrid 是最小可用融合

当前 hybrid 逻辑是：

```text
baseline top_k * 2
vector top_k * 2
合并去重
归一化分数
按 baseline_weight / vector_weight 融合
```

当前已通过 smoke test，不建议 6号窗口调整权重。
