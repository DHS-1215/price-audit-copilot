   四、关键设计决策说明
   决策 1：不直接照搬旧 SQLite 表结构

没有把 cleaned_products / normalized_products / anomaly_details 直接翻译成 MySQL 表，而是抽象为正式业务实体：

product_raw
product_clean
audit_result
rule_definition
rule_chunk
review_task
review_record
ask_log
model_call_log

原因：旧 SQLite 是原型阶段产物，正式数据库必须围绕业务对象，而不是围绕阶段文件名建模。

决策 2：清洗层与结果层明确拆分

将以下字段归入 product_clean：

标准化品牌
规范化规格
干净平台
干净价格
干净标题
干净规格
标题规范提示

将以下字段归入 audit_result：

显式低价阈值
组内均价
当前价格/组均价比
低价规则来源
异常原因

原因：后续规则引擎、解释层、复核层需要基于稳定结果层事实工作，不能把清洗结果和审核结果混成一坨。

决策 3：规则本体与规则解释素材拆分

正式拆为：

rule_definition
rule_chunk

原因：4 号窗口要用规则本体做判定，5 号窗口要用 chunk 做解释，二者职责不同。

决策 4：任务与记录拆分

正式拆为：

review_task
review_record

原因：任务表负责当前状态，记录表负责动作留痕。否则后续 7 号窗口做复核闭环时会非常混乱。

决策 5：ask 日志与模型调用日志拆分

正式拆为：

ask_log
model_call_log

原因：请求级别和模型调用级别粒度不同，后续 trace、性能分析、审计必须分层。

决策 6：旧数据迁移采用“最小可用迁移”

没有在 migration 里塞复杂脏数据转换，而是单独写 scripts/migrate_legacy_sqlite.py。

原因：Alembic 适合做结构迁移和轻量 seed，不适合承载大量历史脏数据清洗。

本窗口明确否掉的方案
否掉“继续让 legacy_sqlite.py 充当未来正式主库实现”
否掉“把旧 SQLite 三张表直接变成正式 MySQL 三张表”
否掉“把历史数据转换硬塞进 Alembic migration”
否掉“2 号窗口顺手把 3 号窗口 config/logger/exception 全写了”
