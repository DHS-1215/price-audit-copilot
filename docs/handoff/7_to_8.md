## 二、本窗口目标回顾


- 异常详情页；

- 复核动作；

- 状态流转；
- 导出能力。

它是基于 4、5、6号窗口已经完成的：

- 结果层；
- 解释层；
- 问答编排层；

补齐人工复核闭环。

## 四、关键设计决策说明

---

### 决策 1：人工复核只基于 `audit_result`，不重新判定异常

7号窗口没有重写：

```text
low_price
cross_platform_gap
spec_risk
retrieval_service
rule_explanation_service
ask_orchestrator
```

原因：

| 能力        | 所属窗口 |
|-----------|------|
| 异常判定      | 4号窗口 |
| RAG 解释    | 5号窗口 |
| `/ask` 编排 | 6号窗口 |
| 人工复核闭环    | 7号窗口 |

7号窗口只负责人工复核闭环。

这样可以避免 UI 层和规则判定层口径打架。

---

### 决策 2：区分 `review_task` 和 `review_record`

当前口径：

```text
review_task   = 一条异常的复核任务
review_record = 对该任务执行过的动作记录
```

一个 `review_task` 可以有多条 `review_record`。

例如：

- 创建任务；
- 添加备注；
- 确认异常；
- 关闭任务；
- 关闭后继续补充备注。

这些都应分别沉淀为 `review_record`。

这样做的好处是：

1. 能还原业务处理过程；
2. 能保留复核历史；
3. 能支撑审计；
4. 能导出最终处理结果。

---

### 决策 3：Repository 和 Service 分层

采用：

```text
API 层
  ↓
Service 层
  ↓
Repository 层
  ↓
ORM / MySQL
```

原因：

- endpoint 不直接写业务逻辑；
- service 负责状态流转和业务判断；
- repository 只负责数据库读写。

这让 8号窗口后续写测试时可以更清晰地测：

- `ReviewService`
- Review API
- 状态机
- 导出结果

---

### 决策 4：状态机由后端强约束，前端只做提示

后端 `ReviewService` 中加入状态流转校验。

当前允许：

```text
pending / processing → confirmed
pending / processing → rejected
pending / processing → ignored
confirmed / rejected / ignored → closed
```

禁止：

```text
confirmed → reject
confirmed → ignore
closed → confirm
closed → reject
closed → ignore
closed → close
```

备注动作永远允许：

```text
add_comment
```

原因：

> 状态变更是业务事实，不能只靠前端按钮禁用。  
> 后端必须兜底，防止接口被直接调用时绕过限制。

---

### 决策 5：工作台复用 `/ask explanation`，不自己解释异常

工作台中的“规则解释与依据”区域调用的是：

```http
POST /ask
```

并传入：

```text
audit_result_id
include_trace = true
```

它复用 6号窗口交接过来的 `/ask explanation` 主链。

工作台展示：

```text
answer / final_summary
rule_facts
evidences
citations
trace
```

原因：

> 7号窗口只负责展示和业务复核，不重新实现解释链。

---

### 决策 6：先做 JSON 导出，不做 Excel 文件导出

当前：

```http
GET /api/v1/reviews/export
```

返回结构化 JSON。

原因：

1. 先保证导出数据结构稳定；
2. 8号窗口可以基于该接口继续补 CSV / Excel；
3. 当前已经满足“可导出复核结果”的基本验收。

## 五、当前统一口径

---

### 5.1 复核任务状态口径

| 状态           | 含义        |
|--------------|-----------|
| `pending`    | 待复核       |
| `processing` | 处理中       |
| `confirmed`  | 已确认异常     |
| `rejected`   | 已判定误报     |
| `ignored`    | 已忽略       |
| `closed`     | 已关闭       |
| `done`       | 已完成，兼容旧口径 |

---

### 5.2 复核动作口径

| 动作                    | 含义                         |
|-----------------------|----------------------------|
| `confirm_abnormal`    | 确认异常                       |
| `mark_false_positive` | 标记误报                       |
| `add_remark`          | 添加备注                       |
| `ignore_task`         | 忽略任务                       |
| `close_task`          | 关闭任务                       |
| `reassign`            | 重新分配，当前 schema 保留，业务暂未正式实现 |

---

### 5.3 审核结果状态口径

当前 7号窗口会更新 `audit_result.result_status`：

| 状态                   | 含义     |
|----------------------|--------|
| `confirmed_abnormal` | 人工确认异常 |
| `false_positive`     | 人工判定误报 |
| `ignored`            | 人工忽略   |
| `pending_review`     | 待复核    |

注意：

> `closed` 只关闭 `review_task`，不一定改变 `audit_result.result_status`。  
> 因为关闭任务更像任务归档，不等同于修改异常事实。

## 八、风险与注意事项

---

### 风险 1：状态机测试会创建新任务

```text
scripts/smoke_test_window7_review_state_machine.py
```

会为了验证状态机创建新任务，并可能关闭历史任务。

8号窗口如果做自动化测试，需要注意测试数据隔离。

建议后续 pytest 阶段使用：

- 测试数据库；
- 独立 `audit_result` 样本；
- 测试后清理数据。

---

### 风险 2：当前导出是 JSON，不是文件流

当前导出接口返回：

```text
ReviewExportResponse
```

不是直接下载 CSV / Excel。

Streamlit 页面当前支持下载 JSON。

如果最终 README 说“支持导出”，应说明当前导出形式是 JSON。

如果要写“Excel 导出”，则需要 8号窗口继续补。

---

### 风险 3：review API 目前还不是统一响应外壳

当前 review endpoint 返回的是 Pydantic response model。

如果项目总口径要求所有接口都统一成：

```json
{
  "success": true,
  "code": "OK",
  "message": "success",
  "trace_id": "...",
  "data": {}
}
```

那么 8号窗口或后续收口需要评估是否统一包装。

但目前不影响 7号窗口闭环验收。

---

### 风险 4：`/ask` 路径不是 `/api/v1/ask`

工作台中规则依据调用的是：

```http
POST /ask
```

不是：

```http
POST /api/v1/ask
```

因为当前实际路由注册结果是：

```text
/ask
/ask-lc
```

8号窗口写文档时必须按真实路径写，不要凭感觉写成 `/api/v1/ask`。

---

### 风险 5：旧乱码数据可能仍残留

中文乱码问题已经通过：

- `charset=utf8mb4`
- MySQL 表 `utf8mb4`
- 中文验收脚本

修复。

但如果数据库里还有修复前写入的 `???` 旧数据，那些旧数据不会自动恢复，需要清理或重建。

当前新数据已验证中文正常。
