# 7号窗口阶段总结：人工复核闭环与业务工作台

## 一、7号窗口定位

7号窗口的目标不是继续做 RAG，也不是继续改 `/ask`，而是把前面已经完成的能力真正串成业务闭环。

前面几个窗口已经完成了：

| 窗口   | 已完成能力                    |
|------|--------------------------|
| 4号窗口 | 异常判定结果层                  |
| 5号窗口 | RAG 检索与规则解释              |
| 6号窗口 | `/ask` 主链与 `/ask-lc` 增强链 |

7号窗口在这个基础上做的是：

```text
异常结果
  ↓
创建复核任务
  ↓
展示异常详情
  ↓
展示规则依据
  ↓
人工确认 / 误报 / 忽略 / 关闭
  ↓
写入复核记录
  ↓
状态流转
  ↓
导出结果
```

一句话：

> 7号窗口负责把系统从“会分析、会解释”，推进到“业务人员能真正处理异常”。

这也是这个项目从 AI Demo 变成业务系统的关键一步。

---

## 二、7号窗口没有做什么

这点面试里要说清楚。

7号窗口没有重新做：

- 低价异常判定；
- 跨平台价差判定；
- 规格风险判定；
- RAG 检索；
- 规则解释服务；
- `/ask` 编排；
- `/ask-lc` Tool Calling。

这些已经是 4、5、6 号窗口的职责。

7号窗口只基于已有的：

```text
audit_result
rule_hit
rule_facts
evidences
citations
```

做：

- 人工复核；
- 状态流转；
- 业务留痕。

这个边界是对的。

如果 7号窗口又重新判定异常，就会造成 UI 层和规则引擎层口径打架。

企业项目里这是大忌。


## 3.2 新增 Service 层

### 文件

```text
app/services/review_service.py
```

这是 7号窗口后端核心。

它负责真正的复核业务逻辑。

### 核心方法

```text
create_task        创建复核任务
list_tasks         查询复核任务
get_task_detail    查询任务详情
confirm_task       确认异常
reject_task        标记误报
ignore_task        忽略任务
close_task         关闭任务
add_comment        添加备注
list_records       查询复核记录
export_tasks       导出复核结果
```

其中最关键的是：

```text
confirm_task
reject_task
ignore_task
close_task
```

它们不是简单写一条记录，而是同时做三件事：

1. 更新 `review_task.task_status`
2. 更新 `audit_result.result_status`
3. 写入 `review_record` 操作记录

所以它形成了真正的业务闭环。

## 五、复核状态机设计

后面补了状态机，防止任务状态乱跳。

当前规则是：

```text
pending / processing → confirmed
pending / processing → rejected
pending / processing → ignored

confirmed / rejected / ignored → closed

closed → 不允许再执行状态动作

add_comment → 任何状态都允许
```

这意味着：

- 待复核任务可以确认异常、标记误报、忽略；
- 已有结论的任务只能关闭；
- 已关闭任务不能再改状态；
- 但任何状态都可以继续补充备注。

这个设计比较稳。

因为真实业务里：

- 备注是留痕行为，可以追加；
- 但状态动作不能随便反复横跳。

## 十、面试可以怎么讲

可以这么说：

> 7号窗口主要解决的是人工复核闭环问题。
>
> 前面系统已经能做异常判定、规则解释和统一问答，但如果业务人员只能看模型回答，项目还只是分析系统。
>
> 所以我在 7号窗口里基于 `audit_result` 建立 `review_task` 和 `review_record` 业务流。
>
> `review_task` 表示一条异常的复核任务，`review_record` 表示业务人员对任务执行过的动作，比如确认异常、标记误报、添加备注、忽略和关闭。
>
> 后端上我补了 `ReviewRepository` 和 `ReviewService`，把数据库读写和业务状态流转分开；接口层提供复核任务创建、列表、详情、确认、误报、备注、关闭和导出接口。
>
> 同时我在 Streamlit 工作台里展示异常详情、审核快照、规则依据、复核记录和操作按钮，并复用 `/ask explanation`
> 主链展示 `rule_facts`、`evidence`、`citation`。
>
> 也就是说，工作台不重新判定异常，而是基于规则引擎和 RAG 解释链的结果做人工确认和留痕。
>
> 最后我补了复核状态机，限制 `pending`、`confirmed`、`closed` 等状态之间的合法流转，避免任务被反复乱改，并写了 smoke test
> 验证复核闭环、中文字段和状态机。

这段可以背熟一点。

技术面问 7号窗口时，这样讲就很清楚。

- 工作台页面加入更细的分页；
- 状态机规则写进 `source_of_truth` 文档；
- 补 review API 文档；
- 补接口测试到 pytest；
- 统一历史接口路径，比如 `/ask` 和 `/api/v1/reviews` 的路径风格。

这些是锦上添花，不影响 7号窗口核心收口。
