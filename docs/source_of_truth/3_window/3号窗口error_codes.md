# 错误码规范（3号窗口正式版）

## 1. 文档目的

本文档用于统一项目基础设施层的错误码口径，避免后续 4~8 号窗口各自定义报错格式，导致接口返回风格分裂、日志难查、前后文档不一致。

本项目错误码的作用不是“显得专业”，而是为了：

- 统一接口失败返回格式
- 统一日志关键字段
- 统一异常处理入口
- 让 ask / model / review 等链路后续具备可追溯性

## 2. 使用原则

### 2.1 错误码是正式口径，不允许随意改名

一旦进入代码与文档，后续窗口不得随意把 `DATABASE_ERROR` 改成别的名字。

### 2.2 错误码和 HTTP 状态码不是一回事

- HTTP 状态码：给客户端和框架层看
- 错误码：给业务链路、日志、审计、文档统一口径用

### 2.3 先保证统一，再逐步细化

当前 3 号窗口先定义基础错误码。  
后续若 4~7 号窗口确有新增业务异常类型，可在不破坏现有口径前提下扩展。

## 3. 当前正式错误码清单

| 错误码                 | 中文说明     | 典型场景                               |
|---------------------|----------|------------------------------------|
| OK                  | success  | 成功响应                               |
| BAD_REQUEST         | 请求参数有误   | 请求参数格式不合法、必要字段缺失                   |
| VALIDATION_ERROR    | 请求校验失败   | FastAPI / Pydantic 校验未通过           |
| NOT_FOUND           | 资源不存在    | 查询目标不存在                            |
| CONFLICT            | 资源冲突     | 状态冲突、重复创建、幂等冲突                     |
| DATABASE_ERROR      | 数据库操作失败  | 数据库连接、查询、写入失败                      |
| MODEL_CALL_ERROR    | 模型调用失败   | Ollama / 模型服务调用异常                  |
| RETRIEVAL_ERROR     | 检索失败     | RAG 检索异常、向量检索异常                    |
| RULE_ENGINE_ERROR   | 规则引擎执行失败 | 清洗 / 判定过程执行异常                      |
| REVIEW_ACTION_ERROR | 复核动作执行失败 | review_task / review_record 相关操作异常 |
| INTERNAL_ERROR      | 服务器内部错误  | 未捕获异常、兜底异常                         |

## 4. 当前异常层次

### 4.1 AppException

项目统一基础异常。

适用：

- 需要统一包装错误码、错误消息、状态码、扩展字段时

### 4.2 BizException

业务异常。

适用：

- 规则判定类异常
- 业务状态冲突
- 业务动作不允许执行

### 4.3 ExternalServiceException

外部依赖异常。

适用：

- 模型调用失败
- 向量库失败
- 第三方服务失败

## 5. 错误响应外壳

失败响应统一采用以下结构：

```json
{
  "success": false,
  "code": "INTERNAL_ERROR",
  "message": "服务器内部错误",
  "trace_id": "xxx",
  "data": null
}
```

字段说明：

| 字段       | 含义                        |
|----------|---------------------------|
| success  | 是否成功，失败时固定为 false         |
| code     | 正式错误码                     |
| message  | 面向调用方的错误说明                |
| trace_id | 链路追踪 ID                   |
| data     | 扩展信息，通常为空；校验异常时可携带 errors |

## 6. 当前约束

### 6.1 不允许直接裸抛字符串错误给接口层

后续窗口应优先使用统一异常类，而不是直接 `raise Exception("xxx")` 作为正式口径。

### 6.2 不允许每个模块自己定义一套错误响应结构

接口失败时统一走全局异常处理，统一输出 `ApiResponse` 外壳。

### 6.3 不允许文档和代码口径脱节

代码中使用的错误码名称必须与本文档一致。

## 7. 后续扩展原则

后续如需扩展错误码，请遵守：

1. 优先复用已有错误码
2. 确有必要再新增
3. 新增后同步更新本文档
4. 不得修改已有错误码的含义

## 8. 一句话总结

错误码不是装饰，而是后续 ask / model / review / rule / retrieval 全链统一报错口径的基础。
