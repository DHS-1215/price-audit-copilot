# 统一响应结构说明（3号窗口正式版）

## 1. 文档目的

本文档用于统一项目接口响应结构口径，避免后续 4~8 号窗口各自返回不同风格的数据，导致接口层混乱、前后端难对接、日志和异常难统一。

3 号窗口当前重点是先把正式响应外壳定住。

## 2. 当前正式响应对象

当前统一响应对象为：

`ApiResponse`

字段如下：

| 字段       | 类型   | 含义        |
|----------|------|-----------|
| success  | bool | 是否成功      |
| code     | str  | 业务成功码或错误码 |
| message  | str  | 响应消息      |
| trace_id | str  | 链路追踪 ID   |
| data     | Any  | 响应数据主体    |

## 3. 成功响应格式

成功响应统一格式：

```json
{
  "success": true,
  "code": "OK",
  "message": "success",
  "trace_id": "xxx",
  "data": {}
}
```

生成方式：

- 使用 `success_response(...)`

## 4. 失败响应格式

失败响应统一格式：

```json
{
  "success": false,
  "code": "INTERNAL_ERROR",
  "message": "服务器内部错误",
  "trace_id": "xxx",
  "data": null
}
```

生成方式：

- 使用 `error_response(...)`
- 或通过全局异常处理器统一生成

## 5. 当前适用范围

### 5.1 当前已适用

- 根接口等新接入接口
- 全局异常失败返回
- 后续新增正式接口

### 5.2 当前暂不强行覆盖

部分历史接口若已存在稳定 `response_model` 合同，可暂时保留。  
例如：

- `/extract` 当前仍保留旧输出合同

原因：  
3 号窗口当前目标是先完成基础设施统一，不是一次性打断所有历史接口。

## 6. 当前设计原则

### 6.1 success / code / message / trace_id / data 为正式外壳字段

后续窗口不应随意增删基础字段。

### 6.2 业务数据统一放入 data

不要把业务字段直接平铺到顶层，避免顶层结构越来越乱。

### 6.3 trace_id 必须保留正式位置

这样接口层、日志层、异常层才真正能串起来。

## 7. 与异常体系的关系

失败响应通常来自：

- AppException
- RequestValidationError
- 未捕获异常兜底

这些都由全局异常处理统一包装为 `ApiResponse` 外壳。

## 8. 与后续窗口的关系

### 8.1 对 4 号窗口

规则引擎相关接口若新增正式 API，应优先使用统一响应外壳。

### 8.2 对 5 号窗口

检索解释相关接口返回时，证据对象建议放在 `data` 内。

### 8.3 对 6 号窗口

`/ask` 主链未来正式收口时，应优先向统一响应外壳靠拢。

### 8.4 对 7 号窗口

review / 导出 / 列表等接口后续统一接这套外壳最稳。

## 9. 当前约束

### 9.1 不允许后续接口各自返回不同顶层格式

正式新接口应优先统一。

### 9.2 不允许错误响应脱离全局异常处理单独乱飞

业务异常优先走统一异常体系。

### 9.3 历史接口收口要渐进，不要暴力一刀切

当前项目仍处于升级过程中，允许旧接口暂时存在，但目标方向必须统一。

## 10. 一句话总结

统一响应结构不是为了好看，而是为了让接口、异常、trace、日志和后续业务链路说同一种话。
