这版设计只做一件事：把你现在“清洗底稿 + 归一底稿 + 旧版异常脚本 + audit_result + rule_definition”收束成正式结果层。因为 4

5 号窗口再围绕这些事实做解释。

一、这次不推翻的东西

这些保留，别乱动：

clean_service.py 里基础字段清洗函数保留，继续做“原子清洗能力”，但不再当主流程脚本。
normalize_service.py 里品牌归一、规格归一、规格来源、规格冲突判断保留，继续做“归一化能力层”。
audit_result 继续保留为“异常结果摘要表”，它已经能承载异常类型、命中规则、规则版本、原因摘要、输入快照、低价类关键数值。
rule_definition 继续保留为“正式规则定义表”，规则编码、版本、启用状态、阈值配置、来源文档都已经有正式落点。
rule_chunk 继续保留给 5 号窗口用，它是规则解释层的数据底盘，不是 4 号窗口的判定明细表。
二、这次必须新增的东西
1）新增模型：app/models/rule_hit.py

这是 4 号窗口最该补的核心。总口径里已经把 rule_hit 定义成“某条数据命中的规则事实”，而现在库里有
rule_definition、rule_chunk、audit_result，唯独缺这层。

建议字段直接定成这版：

id
audit_result_id：外键到 audit_result.id
clean_id：外键到 product_clean.id
anomaly_type：low_price / cross_platform_gap / spec_risk
rule_code
rule_version
rule_definition_id
is_hit
input_snapshot_json
computed_value_json
threshold_snapshot_json
hit_message
hit_order

为什么这样定：因为 audit_result 负责最终摘要，rule_hit 负责命中明细。这样一个 audit_result
可以挂多个命中事实，比如低价异常里同时命中显式规则和统计规则，就不必只靠 low_price_rule_source = both 兜着了。audit_result
现在虽然有 hit_rule_code / hit_rule_version / rule_definition_id / input_snapshot_json，但它更像主命中摘要，不适合承载多命中明细。

2）新增 schema：app/schemas/rule_hit.py

建议拆成：

RuleHitBase
RuleHitCreate
RuleHitRead
RuleHitListItem

字段跟 ORM 对齐就行，不要发明新口径。这样后面 5 号窗口解释层、7 号窗口复核详情页都能直接吃。4 号窗口本来就需要交出 rule hit
留痕设计，这个 schema 是正式交付物之一。

3）新增服务：app/services/rule_engine_service.py

职责只做一件事：纯判定。
不读文件，不写 CSV，不管 API，不管落库，只接收规范化后的数据，产出规则评估结果。4 号窗口本来就要求“把原第二周那套脚本式逻辑升级成正式规则判定系统”，这个文件就是主战场。

4）新增服务：app/services/rule_hit_service.py

职责是把 rule_engine_service 的中间结果整理成 RuleHitCreate 列表。
它不负责最终结果摘要，只负责“命中了什么、怎么算的、为什么命中”。

5）新增服务：app/services/audit_service.py

职责是主流程收束：

接收一批 product_clean
调 rule_engine_service
生成 audit_result
生成 rule_hit
落库
返回结构化结果

这符合总要求里的“清洗 + 异常判定 + /ask”从离散能力升级成统一业务链路。

三、旧 run_analysis() 怎么迁

旧脚本里真正有价值的是这些：

参数常量：MIN_GROUP_SIZE、LOW_PRICE_RATIO_THRESHOLD、CROSS_PLATFORM_GAP_RATIO_THRESHOLD
显式低价规则：EXPLICIT_LOW_PRICE_RULES
统计层：attach_group_stats()
三类判定：detect_low_price_anomaly()、detect_cross_platform_anomaly()、detect_spec_risk()
原因拼装：build_anomaly_reason()

迁移方式不是整文件平移，而是拆成三层：

第一层：归一输入准备

继续复用 clean_service.py 和 normalize_service.py 的能力，先拿到：

normalized_brand
normalized_spec
price
normalized_platform
title_spec_hint
title_spec_mismatch_flag
missing_normalized_spec_flag
第二层：规则引擎计算

把旧脚本里这三类逻辑拆开：

evaluate_low_price(...)

产出：

is_explicit_low_price
is_stat_low_price
explicit_low_price_threshold
group_avg_price
price_to_group_avg_ratio
low_price_rule_source
两条 rule_hit（显式 / 统计）
evaluate_cross_platform_gap(...)

产出：

cross_platform_gap_ratio
是否命中
一条 rule_hit
evaluate_spec_risk(...)

产出：

是否因标题规格冲突命中
是否因规范规格缺失命中
一条或两条 rule_hit
第三层：摘要结果组装

由 audit_service.py 把每个异常类型的判定结果压成一条 audit_result。
建议规则是：每个 clean_id × 每个 anomaly_type 生成一条 audit_result，命中与否都可以保留，这样后续可追溯性最好。

四、规则配置怎么补细

你现在 seed 里已经有 4 条规则：

LOW_PRICE_EXPLICIT
LOW_PRICE_STAT
CROSS_PLATFORM_GAP
SPEC_RISK

方向是对的，但 threshold_config_json 还不够可执行，要补成这种粒度。

LOW_PRICE_EXPLICIT

现在只有 examples，不够机器可执行。建议改成：

{
"mode": "explicit",
"rules": [
{"brand": "鸿茅", "spec": "500ml", "threshold": 180.0},
{"brand": "鸿茅", "spec": "500ml*4瓶", "threshold": 799.0}
]
}

这就是把旧脚本里的 EXPLICIT_LOW_PRICE_RULES 正式迁出来。

LOW_PRICE_STAT

现在只有 compare_field，不够。建议补成：

{
"mode": "stat",
"min_group_size": 3,
"ratio_threshold": 0.8,
"compare_field": "price_to_group_avg_ratio"
}

这就是把旧脚本里的 MIN_GROUP_SIZE 和 LOW_PRICE_RATIO_THRESHOLD 迁出来。

CROSS_PLATFORM_GAP

现在只有 mode: gap，不够。建议补成：

{
"mode": "gap",
"min_distinct_platforms": 2,
"gap_ratio_threshold": 0.25,
"flag_only_lowest_price": true
}

你旧脚本里这条逻辑最需要收紧，因为它现在按组样本数判，不是按不同平台数判，而且默认只标最低价那条记录，这些都应该进规则配置和规则说明，不要藏在代码里。

SPEC_RISK

现在只有 mode: parse_risk，建议补成：

{
"mode": "spec_risk",
"conditions": ["title_spec_mismatch", "missing_normalized_spec"]
}

这样后面你既能保留一个总规则 SPEC_RISK，也能在 rule_hit 里记录具体命中原因。

五、几个关键口径，先统一
1）rule_type 统一成三种

你现在种子规则里用的是：

low_price
cross_platform_gap
spec_risk

而 RuleDefinition 注释和 schema 注释里还残留 gap 这种说法。这个别留了，统一成：

low_price
cross_platform_gap
spec_risk
不然 4 号窗口和 5 号窗口之间迟早打架。
2）正式层只用英文字段

总要求已经定了：表名、字段名统一英文蛇形，中文字段只用于展示、导出、文档说明。旧脚本里大量中文列名可以继续作为过渡和 CSV
验收用，但新 service / ORM / schema 不再继续扩中文主字段。

3）新代码必须接 3 号窗口基础设施

3 号窗口已经明确了：4 号窗口直接复用统一 logger / trace / exception，不要自己重造一套。新增正式接口和正式 service
都应优先走统一异常和统一响应外壳。

六、audit_service.py 主流程，我建议就这么走
def run_audit_for_clean_records(clean_records: list[ProductClean]) -> AuditBatchResult:
# 1. 预加载启用中的 rule_definition
# 2. 按异常类型分组规则
# 3. 先做统计底座（group stats / platform stats）
# 4. 对每条 clean_record 分别评估：
# - low_price
# - cross_platform_gap
# - spec_risk
# 5. 生成 rule_hit 列表
# 6. 按 anomaly_type 生成 audit_result 摘要
# 7. 批量写入 audit_result 和 rule_hit
# 8. 返回批次结果

这样做的好处是：4 号窗口交出去的就是正式结果层事实，5 号窗口只用解释这些事实，不用再去猜你是怎么判的。
