这版设计只做一件事：

> 把当前“清洗底稿 + 归一底稿 + 旧版异常脚本 + `audit_result` + `rule_definition`”收束成正式结果层。

因为 4 号窗口本来就该完成：

- 清洗服务层；
- 判定服务层；
- 规则配置结构；
- `rule_hit` 留痕；
- `audit_result` 写入逻辑；
- 结果字段文档。

先把“系统到底怎么判异常”钉死，后面的 5 号窗口再围绕这些事实做解释。

## 二、这次必须新增的东西

---

## 2.1 新增模型：`app/models/rule_hit.py`

这是 4 号窗口最该补的核心。

总口径里已经把 `rule_hit` 定义成：

> 某条数据命中的规则事实。

而现在库里有：

- `rule_definition`
- `rule_chunk`
- `audit_result`

唯独缺这层。

---

### 建议字段

建议字段直接定成这版：

```text
id
audit_result_id
clean_id
anomaly_type
rule_code
rule_version
rule_definition_id
is_hit
input_snapshot_json
computed_value_json
threshold_snapshot_json
hit_message
hit_order
```

---

### 字段说明

| 字段                        | 说明                                                    |
|---------------------------|-------------------------------------------------------|
| `id`                      | 主键                                                    |
| `audit_result_id`         | 外键到 `audit_result.id`                                 |
| `clean_id`                | 外键到 `product_clean.id`                                |
| `anomaly_type`            | 异常类型：`low_price` / `cross_platform_gap` / `spec_risk` |
| `rule_code`               | 命中的规则编码                                               |
| `rule_version`            | 命中的规则版本                                               |
| `rule_definition_id`      | 外键到 `rule_definition.id`                              |
| `is_hit`                  | 是否命中                                                  |
| `input_snapshot_json`     | 输入快照                                                  |
| `computed_value_json`     | 计算结果快照                                                |
| `threshold_snapshot_json` | 阈值配置快照                                                |
| `hit_message`             | 命中说明                                                  |
| `hit_order`               | 命中顺序                                                  |

---

### 为什么这样定

因为：

- `audit_result` 负责最终摘要；
- `rule_hit` 负责命中明细。

这样一个 `audit_result` 可以挂多个命中事实。

例如：

> 低价异常里同时命中显式规则和统计规则。

这时候就不必只靠：

```text
low_price_rule_source = both
```

兜着了。

`audit_result` 现在虽然有：

- `hit_rule_code`
- `hit_rule_version`
- `rule_definition_id`
- `input_snapshot_json`

但它更像主命中摘要，不适合承载多命中明细。

## 2.3 新增服务：`app/services/rule_engine_service.py`

职责只做一件事：

> 纯判定。

它不负责：

- 读文件；
- 写 CSV；
- 管 API；
- 管落库。

它只接收规范化后的数据，产出规则评估结果。

4 号窗口本来就要求：

> 把原第二周那套脚本式逻辑升级成正式规则判定系统。

这个文件就是主战场。

---

## 2.4 新增服务：`app/services/rule_hit_service.py`

职责是：

> 把 `rule_engine_service` 的中间结果整理成 `RuleHitCreate` 列表。

它不负责最终结果摘要。

它只负责：

- 命中了什么；
- 怎么算的；
- 为什么命中。

---

## 2.5 新增服务：`app/services/audit_service.py`

职责是主流程收束。

流程如下：

```text
接收一批 product_clean
        ↓
调用 rule_engine_service
        ↓
生成 audit_result
        ↓
生成 rule_hit
        ↓
落库
        ↓
返回结构化结果
```

这符合总要求里的：

> 清洗 + 异常判定 + `/ask` 从离散能力升级成统一业务链路。

## 三、旧 `run_analysis()` 怎么迁

旧脚本里真正有价值的是这些：

- 参数常量：
    - `MIN_GROUP_SIZE`
    - `LOW_PRICE_RATIO_THRESHOLD`
    - `CROSS_PLATFORM_GAP_RATIO_THRESHOLD`
- 显式低价规则：
    - `EXPLICIT_LOW_PRICE_RULES`
- 统计层：
    - `attach_group_stats()`
- 三类判定：
    - `detect_low_price_anomaly()`
    - `detect_cross_platform_anomaly()`
    - `detect_spec_risk()`
- 原因拼装：
    - `build_anomaly_reason()`

迁移方式不是整文件平移，而是拆成三层。

---

## 3.2 第二层：规则引擎计算

把旧脚本里的三类逻辑拆开。

### 3.2.1 `evaluate_low_price(...)`

产出：

命中与否都可以保留。

这样后续可追溯性最好。

## 四、规则配置怎么补细

你现在 seed 里已经有 4 条规则：

```text
LOW_PRICE_EXPLICIT
LOW_PRICE_STAT
CROSS_PLATFORM_GAP
SPEC_RISK
```

方向是对的。

但 `threshold_config_json` 还不够可执行，要补成下面这种粒度。

---

## 4.1 `LOW_PRICE_EXPLICIT`

现在只有 examples，不够机器可执行。

建议改成：

```json
{
  "mode": "explicit",
  "rules": [
    {
      "brand": "鸿茅",
      "spec": "500ml",
      "threshold": 180.0
    },
    {
      "brand": "鸿茅",
      "spec": "500ml*4瓶",
      "threshold": 799.0
    }
  ]
}
```

这就是把旧脚本里的 `EXPLICIT_LOW_PRICE_RULES` 正式迁出来。

---

## 4.2 `LOW_PRICE_STAT`

现在只有 `compare_field`，不够。

建议补成：

```json
{
  "mode": "stat",
  "min_group_size": 3,
  "ratio_threshold": 0.8,
  "compare_field": "price_to_group_avg_ratio"
}
```

这就是把旧脚本里的：

- `MIN_GROUP_SIZE`
- `LOW_PRICE_RATIO_THRESHOLD`

迁出来。

---

## 4.3 `CROSS_PLATFORM_GAP`

现在只有：

```json
{
  "mode": "gap"
}
```

不够。

建议补成：

```json
{
  "mode": "gap",
  "min_distinct_platforms": 2,
  "gap_ratio_threshold": 0.25,
  "flag_only_lowest_price": true
}
```

这条逻辑最需要收紧。

因为旧脚本里它现在是：

- 按组样本数判；
- 不是按不同平台数判；
- 默认只标最低价那条记录。

这些都应该进规则配置和规则说明，不要藏在代码里。

---

## 4.4 `SPEC_RISK`

现在只有：

```json
{
  "mode": "parse_risk"
}
```

建议补成：

```json
{
  "mode": "spec_risk",
  "conditions": [
    "title_spec_mismatch",
    "missing_normalized_spec"
  ]
}
```

这样后面既能保留一个总规则 `SPEC_RISK`，也能在 `rule_hit` 里记录具体命中原因。

## 六、`audit_service.py` 主流程建议

建议主流程设计成：

```python
def run_audit_for_clean_records(clean_records: list[ProductClean]) -> AuditBatchResult:
    ...
```

---

### 主流程步骤

1. 预加载启用中的 `rule_definition`
2. 按异常类型分组规则
3. 先做统计底座：
    - group stats
    - platform stats
4. 对每条 `clean_record` 分别评估：
    - `low_price`
    - `cross_platform_gap`
    - `spec_risk`
5. 生成 `rule_hit` 列表
6. 按 `anomaly_type` 生成 `audit_result` 摘要
7. 批量写入 `audit_result` 和 `rule_hit`
8. 返回批次结果

---

### 这样做的好处

4 号窗口交出去的就是正式结果层事实。

5 号窗口只用解释这些事实，不用再去猜你是怎么判的。
