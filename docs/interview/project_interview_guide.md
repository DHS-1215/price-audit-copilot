# 项目面试讲解文档

## 一、项目一句话介绍

这是一个面向电商维价审核场景的 AI 审核辅助系统原型，覆盖商品数据清洗、异常判定、规则解释、统一问答编排、人工复核闭环、日志审计和 Docker 部署交付。

它不是通用聊天机器人，而是一个围绕真实业务流程设计的 AI 业务系统。

---

## 二、项目解决什么问题

在电商维价业务中，业务人员需要持续检查多个平台上的商品价格、规格、标题和店铺信息，判断是否存在低价、价差、规格识别风险等异常。

传统处理方式主要依赖：

- 人工看表；
- 人工比对；
- 人工查规则；
- 人工判断异常原因。

这种方式效率低，而且很难解释每条异常为什么被判定。

所以我做了这个系统，目标是：

```text
自动清洗商品数据
自动判定异常
给出规则依据
支持自然语言查询
支持人工复核和留痕
最后能够测试、部署和交付
```

## 六、为什么这个项目不是普通 RAG Demo

普通 RAG Demo 通常是：

```text
上传文档
  ↓
用户提问
  ↓
模型检索
  ↓
模型回答
```

但我这个项目不是单纯文档问答，它有明确业务主链路：

```text
数据清洗
  ↓
异常判定
  ↓
规则命中
  ↓
规则解释
  ↓
人工复核
  ↓
留痕导出
```

RAG 在这里不是核心业务本身，而是给异常判定结果提供可追溯解释。

也就是说：

```text
规则引擎负责判断事实；
RAG 负责解释依据；
模型不能反过来推翻业务结果。
```

---

## 七、为什么要先做规则引擎，再做 RAG

因为这个项目的业务核心是“维价审核”，不是闲聊问答。

异常是否成立，应该由规则和数据决定。

例如：

- 是否低于显式低价阈值；
- 是否明显低于组内均价；
- 是否存在跨平台价差；
- 标题规格和结构化规格是否不一致。

这些判断必须可控、可追溯，不能交给大模型自由发挥。

所以我先做规则引擎，生成确定性的异常结果；再让 RAG 去找规则文档，解释为什么这个结果成立。

这样可以避免：

```text
数据库判定一种结果，
模型解释成另一种结果。
```

这也是企业级项目里非常关键的边界。

## 八、为什么 `/ask` 不直接交给自由 Agent

我做过区分：

```text
/ask    是正式主链
/ask-lc 是 LangChain 增强链
```

正式主链不能完全交给自由 Agent，因为业务审核系统更看重：

- 稳定；
- 可控；
- 可验收；
- 可追踪。

如果全部交给 Agent，它可能会：

- 乱选工具；
- 跳过规则依据；
- 改写问题；
- 生成看似合理但不可追溯的答案。

所以我把问题分成：

```text
analysis
retrieval
explanation
mixed
```

然后由系统受控编排不同工具。

这样做的好处是：

- 每类问题走固定链路；
- 输出结构稳定；
- trace 可追踪；
- 方便测试验收。

---

## 九、`/ask` 和 `/ask-lc` 为什么同时保留

`/ask` 是正式主链，强调：

- 稳定；
- 可控；
- 可验收。

`/ask-lc` 是增强链，用于展示 LangChain tool calling 能力。

我没有让 `/ask-lc` 替代 `/ask`，是因为在业务系统里，稳定主链更重要。

可以这样讲：

```text
/ask 负责生产级主流程；
/ask-lc 负责展示工具调用和 Agent 化增强能力；
两者并存，但职责不同。
```

这比“所有东西都交给 Agent”更稳。

## 十、RAG 是怎么做的

RAG 这块我主要做了几件事：

1. 规则文档组织；
2. Markdown 规则文档切分；
3. chunk metadata 构建；
4. `rule_chunk` 入库；
5. baseline 检索；
6. vector 检索；
7. hybrid 检索；
8. evidence / citation 输出。

重点不是只“搜出来”，而是每条证据要能追到：

- 哪份规则文档；
- 哪个 chunk；
- 哪个章节；
- 为什么命中；
- 对应什么规则类型。

比如解释规格风险时，可以返回具体证据和 citation，而不是只让模型说：

> 我觉得这里有风险。

## 十一、baseline / vector / hybrid 检索怎么讲

### 11.1 baseline 检索

baseline 检索偏规则打分，适合规则文档这种结构化文本。

它更强调：

- 规则码；
- 异常类型；
- 标题；
- 章节；
- 关键词；
- metadata。

---

### 11.2 vector 检索

vector 检索偏语义匹配，适合用户自然语言表达和文档表述不完全一致的情况。

它更适合补足：

- 同义表达；
- 长尾问法；
- 非标准提问。

---

### 11.3 hybrid 检索

hybrid 检索结合两者：

```text
baseline 保证业务关键词和规则结构命中；
vector 补充语义召回；
hybrid 让结果更稳。
```

这个项目里，我保留 baseline，不是因为不会向量检索，而是因为规则类业务不能完全依赖语义相似度。

这句话很重要，面试官一听就知道你没乱用 RAG。

## 十二、人工复核闭环怎么讲

人工复核是这个项目从“分析系统”升级成“业务系统”的关键。

复核链路是：

```text
audit_result
  ↓
review_task
  ↓
review_record
  ↓
状态流转
  ↓
导出
```

业务人员可以：

- 查看异常；
- 查看详情；
- 查看规则依据；
- 添加备注；
- 确认异常；
- 标记误报；
- 忽略任务；
- 关闭任务；
- 查询复核记录；
- 导出结果。

我还做了状态机限制，比如：

```text
pending 可以 confirm
confirmed 不允许再次 confirm / reject / ignore
confirmed 可以 close
closed 不允许状态动作
closed 仍允许 add_comment
```

这样系统不是只让人“看”，而是能让人“处理”。

## 十六、项目难点怎么讲

---

### 难点一：规则结果和模型解释不能打架

解决方式：

```text
先由规则引擎生成 audit_result 和 rule_hit；
RAG 只围绕结果层事实找依据；
模型解释必须服从数据库结果。
```

---

### 难点二：Agent 自由度和业务稳定性冲突

解决方式：

```text
正式主链使用受控编排；
LangChain 只作为增强链保留；
mixed 问题也不完全交给自由 Agent。
```

---

### 难点三：复核状态流转不能随便改

解决方式：

```text
设计 review_task / review_record；
通过状态机限制动作；
用 smoke test 和 pytest 验证状态流转。
```

---

### 难点四：项目从原型升级为可交付工程

解决方式：

```text
补工程分层；
补数据库迁移；
补配置中心；
补日志和 trace；
补测试；
补 Docker；
补部署文档和测试文档。
```

## 十九、面试官可能问：项目怎么从用户输入到输出

可以这样讲：

> 如果用户问的是统计类问题，`/ask` 会走 analysis 路由，查询数据库中的审核结果并返回统计。
>
> 如果问的是规则类问题，会走 retrieval 路由，调用 RAG 检索规则文档。
>
> 如果问的是某条异常为什么命中，会走 explanation 路由，先查 `audit_result` 和 `rule_hit`，再检索规则依据，最后生成解释。
>
> 如果是综合问题，会走 mixed 路由，按受控流程组合分析、检索和报告工具。

一句话总结：

```text
先路由，再调工具，再组织结构化结果，不是所有问题都直接丢给模型。
```

## 二十、面试官可能问：数据怎么进入系统

可以这样答：

> 在真实业务里，商品数据通常来自 RPA、爬虫或业务系统导出的表格。
>
> 在我的项目里，数据先进入 `product_raw`，再经过清洗和标准化进入 `product_clean`，然后由规则引擎生成 `audit_result` 和 `rule_hit`。

继续说：

> 也就是说，大模型不是直接面对原始脏数据，而是基于清洗、判定、落库后的结果做解释和问答。

这能体现你不是只会调模型，而是懂数据链路。

## 二十一、面试官可能问：为什么要做人工复核

> 因为价格异常审核不是模型说了就结束，业务人员最终还要判断这条异常是否确认、是否误报、是否需要忽略或关闭。
