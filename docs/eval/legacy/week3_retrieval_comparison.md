# 第三周检索方案对比总结

## 一、对比目标

第三周在规则知识库检索部分，已经形成了两套方案：

1. baseline 检索
2. 向量检索（FAISS + Ollama embeddings）

本次对比的目的不是单纯比较“谁分数更高”，而是确认：

- 哪类问题 baseline 更稳
- 哪类问题向量检索更强
- 当前两套方案各自的优势和局限
- 第四周是否需要走 hybrid 检索路线

---

## 二、当前检索基础

第三周规则知识库已经完成文档 ingest，当前规则库规模为：

- 规则文档数量：6
- chunk 数量：43

当前规则文档包括：

- 平台价格审核规则
- 规格归一与规格风险识别规则
- 低价异常判定规则
- 跨平台价差异常判定规则
- 人工复核流程说明
- FAQ

---

## 三、方案一：baseline 检索

### 1. 当前实现方式

baseline 检索器当前采用：

- 业务路由
- 关键词提取
- 可解释打分
- FAQ 问句偏好
- 主规则文档与 FAQ 的优先级区分

对应模块：

- `app/rag/retriever.py`

### 2. baseline 的优点

#### （1）结果可控、可解释
baseline 检索的每一条结果都有打分原因，可以清楚知道：

- 是命中了优先文档
- 还是命中了章节标题
- 还是命中了正文关键词

这一点对于调试非常有价值。

#### （2）对业务明确问题更稳
对于问题里已经包含明确业务词的情况，例如：

- 低价
- 跨平台价差
- 规格
- 人工复核

baseline 往往能更稳定地把主规则文档拉进前几名。

#### （3）更容易与结果层对齐
第三周解释链要求先尊重第二周结果层，尤其低价解释必须先看：

- `低价规则来源`
- `组内均价`
- `当前价格/组均价比`

baseline 检索由于路由和关键词都更可控，因此更适合先做规则解释链的第一版。

### 3. baseline 当前表现

从第三周测试结果看：

- 在“为什么这个商品会被判成疑似异常低价？”这个问题上，baseline 检索能把《疑似异常低价判定规则说明》排到前面，FAQ 作为补充进入 top3。
- 在“人工复核时应该先看什么？”这个问题上，baseline 检索能稳定命中《人工复核流程说明》。
- 在“如果标题不完整，规则上该怎么处理？”这类问句上，FAQ 往往更容易排在第一，但主规格规则文档也能进入 top3。

### 4. baseline 的局限

#### （1）对表述变化不够鲁棒
如果用户换一种问法，但没有直接命中预设关键词，baseline 召回质量会下降。

#### （2）容易命中 FAQ 或泛章节
比如：

- FAQ 问句式章节
- “文档目的”
- “业务含义说明”

这些章节有时会排得过前，虽然不算错，但不一定是最强依据。

#### （3）更依赖人工调权重
当前 baseline 的效果很大程度上建立在：

- 路由词表
- 关键词词表
- FAQ 偏好
- 主文档优先权重

这些规则后续仍然需要手动维护。

## 四、方案二：向量检索（FAISS）

### 1. 当前实现方式

向量检索当前采用：

- Ollama embedding 模型：`qwen3-embedding`
- 本地向量索引：FAISS
- 文档向量持久化：`data/faiss/`

对应模块：

- `app/rag/faiss_store.py`
- `app/rag/faiss_retriever.py`

### 2. 向量检索的优点

#### （1）更强的语义召回能力
向量检索不完全依赖用户问题是否出现精准关键词。
即使用户换一种更自然的表达，也有机会召回相关规则 chunk。

#### （2）对 FAQ 问句的自然匹配更强
像下面这种问题：

- 为什么这个商品会被判成疑似异常低价？
- 跨平台价差异常是怎么判的？
- 如果标题不完整，规则上该怎么处理？

向量检索往往会先把 FAQ 里语义最接近的问题排上来，这对用户体验是有帮助的。

#### （3）在规格和标题不完整这类语义型问题上更有优势
从第三周测试结果看，像“如果标题不完整，规则上该怎么处理？”这类问题，向量检索能比较自然地命中：

- FAQ 相关问句
- “规格列优先，但允许参考标题提示”
- “归一规格缺失”

这说明它已经具备一定语义召回能力。

### 3. 向量检索当前表现

从第三周测试结果看：

- 低价问题时，FAQ 往往排在第一，主规则文档排在后面。
- 跨平台问题时，FAQ 和主规则文档都能进入 top3。
- 规格问题时，FAQ 问句和规格正文都能被召回。
- 人工复核问题时，基本能稳定回到《人工复核流程说明》。

### 4. 向量检索的局限

#### （1）FAQ 容易排得太前
由于 FAQ 本身更接近用户自然语言问句，向量检索里 FAQ 往往更容易排到第一。
这对“回答自然性”有帮助，但不一定最适合做“正式规则依据”。

#### （2）当前还缺少二次重排
现在向量检索主要是“embedding 相似度直接返回 top-k”，还没有再做：

- FAQ 降权
- 主规则文档优先
- 泛章节降权
- 核心逻辑章节加权

所以目前它的结果更像“语义召回结果”，还不是最终最优答案排序。

#### （3）可解释性弱于 baseline
向量检索现在能返回相似度分数，但不像 baseline 一样能明确说出：

- 为什么命中这个结果
- 是哪个业务词触发的
- 为什么这条比那条高

所以从调试角度看，baseline 仍然更透明。

## 五、当前对比结论

### 1. baseline 更适合做什么

当前更适合用于：

- 规则解释链第一版
- 结果层字段强约束场景
- 需要可解释调试的场景
- 问题类型比较明确的场景

尤其在第三周这个阶段，baseline 更像“稳妥、可控的业务检索器”。

### 2. 向量检索更适合做什么

当前更适合用于：

- 用户自然语言问法变化较大的问题
- FAQ 问句召回
- 规格 / 标题不完整这类更偏语义理解的问题
- 给规则检索提供补充召回

### 3. 当前两者并不是替代关系

第三周的 baseline 和向量检索不是互相替代，而是更适合看成：

- baseline：规则型精确召回
- vector：语义型补充召回

也就是说：

> baseline 更像“稳”，vector 更像“宽”。

## 六、为什么第四周应该考虑 hybrid 检索

如果第四周直接只保留 baseline，会有一个问题：

- 对表述变化的适应性不足

如果第四周直接只保留向量检索，也会有一个问题：

- FAQ 和泛章节可能排得过前
- 对正式规则依据的控制力不足

因此更合理的方向不是二选一，而是：

> hybrid 检索

即：

1. baseline 检索先召回一批结果
2. 向量检索再召回一批结果
3. 合并候选结果
4. 做一轮重排
5. 返回最终 top-k

---

## 七、第四周推荐的 hybrid 检索方向

### 1. 召回阶段
同时跑两路：

- 路线 A：baseline 检索
- 路线 B：FAISS 向量检索

### 2. 合并阶段
把两路结果按 `chunk_id` 去重后合并。

### 3. 重排阶段
优先级建议：

1. 主规则文档优先于 FAQ
2. 具体规则章节优先于“文档目的”
3. 包含“逻辑 / 阈值 / 字段说明 / 步骤 / 核心场景”的章节加权
4. FAQ 作为补充证据保留 1 条即可
5. 最终返回 top3

### 4. 输出阶段
上层解释链优先引用：

- 主规则文档作为正式依据
- FAQ 作为补充说明

这样既保留了语义召回能力，又不会让 FAQ 喧宾夺主。
